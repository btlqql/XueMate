# C4 网络技术挑战赛 A 组定位：XueMate 参赛包装

> 结论：XueMate 应该报 **中国高校计算机大赛-网络技术挑战赛 创意（A）普通 A 赛项**。不要把作品讲成单纯“小学生聊天软件”，而要讲成：**面向小学学习场景的云边协同智能学习网络平台**。

## 1. 比赛调性

根据 2026 年竞赛规程，网络技术挑战赛分为创意（A）和攻关（B）两大系列。创意（A）系列包括：

- 普通 A：不限领域与主题。
- A-ST：网络空间与信息安全专项。
- A-ICV：车联网与交通智能网联专项。

普通 A 作品虽然不限领域，但选题需要至少关联网络技术分支之一，例如：

- 网络智能
- 软件定义网络
- 云计算
- 边缘计算
- IPv6 网络
- 物联网
- 工业互联网
- 车联网
- 算力网络
- 网络安全
- 网络数据分析

也可以是：把网络与 IT 技术用于解决经济社会各领域关注的应用热点或难点问题，并鼓励交叉融合。

官方评审口径：

| 阶段 | 创意 | 技术 | 应用 | 设计 | 效果 |
|---|---:|---:|---:|---:|---:|
| 资格赛 | 40% | 30% | 30% | - | - |
| 选拔赛 | 20% | 30% | 20% | 30% | - |
| 挑战赛 | 20% | 20% | 20% | 20% | 20% |

所以 XueMate 答辩必须同时覆盖：创新点、技术深度、应用价值、系统设计、实际效果。

## 2. 往届优秀作品共同特征

从 2025、2021 等往届获奖作品看，A 组优秀作品通常有这些特征：

1. **不是单点功能，而是完整系统**
   - 例如源码静态分析与智能修复系统、变电站智能安防系统、执法案件分析与决策系统、Agent 会议助手、SFC 优化调度系统。

2. **有明确应用场景**
   - 工业质检、交通感知、安防巡检、执法辅助、医疗诊断、教育实验平台、智慧农业、智慧校园等。

3. **技术名词很“硬”**
   - 大模型、深度学习、边缘计算、IoT、LoRaWAN、P4、SDN、DRL、知识图谱、云边协同、网络数据分析。

4. **作品名称偏“行业解决方案”**
   - 常见格式：`品牌名—基于xxx的xxx系统`。
   - 比如：`精雕细“啄”-面向网络安全的源码静态分析与智能陷修复系统`。

5. **必须能现场演示**
   - 决赛阶段重视系统/实物现场展示与答辩，所以 PPT 讲概念不够，必须打开软件展示闭环。

## 3. XueMate 的正确参赛定位

不要说：

```txt
我们做了一个小学生 AI 聊天助手。
```

也不要说：

```txt
我们做了一个教育软件。
```

应该说：

```txt
XueMate 是面向小学学习场景的云边协同智能学习网络平台。
它通过边缘端本地知识库、云端多模态模型、网络学习资源检索、DOM/视觉网页智能体和长期学习画像，构建从资料导入、个性化问答、网页资料获取、动画讲解到复习追踪的完整学习闭环。
```

一句话版本：

```txt
XueMate 用网络智能和云边协同技术，把分散的课件、网页和学生学习行为连接成一个可持续进化的小学学习支持网络。
```

## 4. 和 A 组网络技术的对应关系

| 比赛关注的网络技术方向 | XueMate 对应能力 | 答辩表达 |
|---|---|---|
| 网络智能 | 多模态网页智能体、DOM 辅助决策 | 模型不仅回答问题，还能理解网页并执行资料检索任务 |
| 云计算/算力网络 | 云端大模型 API + 本地 Electron 端 | 端侧负责数据与交互，云侧负责复杂推理 |
| 边缘计算 | 本地 SQLite、RAG、记忆系统 | 学生资料和学习画像优先本地处理，降低隐私风险和网络依赖 |
| 网络数据分析 | 网页搜索、网页内容抽取、资料结构化 | 把开放网络资源转化为小学生可理解的学习材料 |
| 物联网/家庭学习终端延展 | 未来可接入平板、课堂大屏、学习机 | 从单机助手扩展为家庭/课堂学习网络节点 |
| 交叉融合 | 网络技术 + AI + 小学教育 | 对接教育数字化与个性化学习痛点 |

## 5. 建议作品名称

首选：

```txt
XueMate 学伴网络——面向小学学习场景的云边协同智能学习助手平台
```

更技术化：

```txt
XueMate：基于网络智能与边缘 RAG 的小学生个性化学习支持平台
```

更比赛风：

```txt
童学云伴——面向小学教育的云边协同多模态学习网络平台
```

不建议：

```txt
小学生 AI 助手
智能聊天学习软件
儿童版 ChatGPT
```

这些名字太轻，网络技术挑战赛评委会觉得技术含量不够。

## 6. 答辩主线

### 背景痛点

小学学习资源高度分散：

- 老师课件在本地文件里。
- 作业要求在不同文档里。
- 网络资料质量参差不齐。
- 学生薄弱点缺少长期追踪。
- 家长和老师难以持续个性化辅导。

### 技术方案

XueMate 构建了一个“端侧学习智能体 + 云端多模态推理 + 网络资源采集 + 长期学习画像”的系统：

```txt
本地资料/网页资源/学生行为
        ↓
文档解析 + Hybrid RAG + 网页抽取
        ↓
云端 LLM / 多模态模型推理
        ↓
动画讲解 / 作业提示 / 复习队列
        ↓
Memory Atom 长期学习画像
```

### 核心创新

1. **边缘 RAG 学习知识库**
   - 学生资料本地化处理。
   - Hybrid RAG 结合向量、关键词、章节结构和 MMR 多样性。

2. **网络资源智能采集**
   - 支持网页搜索、内容抽取和学习资料辅助整理。
   - Computer Use 通过实时网页 + 多模态截图 + DOM 候选完成资料检索。

3. **长期学习画像**
   - Memory Atom 记录薄弱点、强项、误区、目标和行为证据。
   - 自动形成复习队列。

4. **个性化学习知识图谱**
   - 使用 Sigma.js + Graphology 构建类似 Obsidian Graph 的学习网络。
   - 把资料、知识点、RAG 片段、学习画像和复习任务连接起来。
   - 支持节点搜索、类型过滤和证据查看，增强 RAG 与 Memory 的可解释性。

5. **小学生友好表达**
   - 动画组件替代复杂公式/抽象文字。
   - 回答不直接代写作业，而是给提示和引导。

### 演示闭环

```txt
导入老师课件
→ 学生提问
→ RAG 引用资料回答
→ 动画解释难点
→ 自动搜索网页资料
→ 记录薄弱点
→ 生成复习队列
```

## 7. 需要补强的地方

为了更像 A 组优秀作品，后续应优先补强：

1. **网络资源质量评分**
   - 对搜索到的网页做适龄性、可信度、可读性评分。
   - 这是“网络数据分析”的显性亮点。

2. **学习资源流转可视化**
   - 显示从网页/课件到 RAG chunk、引用、回答、复习队列的数据流。
   - 评委更容易看懂系统复杂度。

3. **教师/家长端概念**
   - 不一定马上做完整后台，但 PPT 里要画架构：学生端、家长端、教师端、云端模型服务。

4. **效果指标统一写入文档**
   - RAG 命中率提升。
   - 上下文压缩率。
   - 记忆去重率。
   - 网页资料抽取成功率。
   - 动画讲解任务完成率。

5. **作品名称和 UI 术语去儿童化但不幼稚**
   - 面向小学生，但评审是计算机专家。
   - UI 可以亲和，答辩术语必须专业。

## 8. 当前项目可直接支撑的技术点

当前代码已经有这些可以讲的模块：

- Electron + Vue 桌面端。
- SQLite 本地数据存储。
- 文档解析与知识库导入。
- Hybrid RAG 检索。
- Memory Atom 长期记忆。
- Sigma.js + Graphology 学习网络。
- 多模态 Computer Use。
- 实时 BrowserView 网页区域。
- DOM 候选抽取。
- 动画组件渲染。
- RAG 与 Memory benchmark 数据。

这些已经足够支撑“可运行系统”的基本盘。下一步重点不是继续堆入口，而是把系统包装成一条清楚的参赛主线。

## 9. 推荐最终口径

```txt
XueMate 是面向小学学习场景的云边协同智能学习网络平台。
系统以学生本地资料和开放网络学习资源为输入，通过边缘端 Hybrid RAG、云端多模态推理、DOM/视觉网页智能体和 Memory Atom 长期学习画像，实现资料问答、作业辅导、动画讲解、网页资料辅助和复习追踪的一体化闭环。
它不是单纯聊天机器人，而是一个把“本地学习资料、开放网络资源、学生学习行为”连接起来的网络智能学习系统。
```
